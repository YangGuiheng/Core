一、概述
1.1 定义
本规范定义Core设备内置EEPRAM阵列的硬件规格、数据记录格式、写入策略、生命周期管理及更换流程。该EEPRAM阵列作为Core设备的“飞行记录仪”，持续记录设备运行数据，用于故障诊断、硬件健康监测和安全事件追溯。

1.2 核心定位
属性	说明
设备名称	Core EEPRAM 阵列（飞行记录仪）
总容量	18GB（16×1GB + 2GB ECC）
写入频率	每小时一次
保留周期	10年
写入权限	仅 CoreOS / Core RTOS 系统级调用可写入
读取权限	除系统级调用外，一切修改、读取行为均被禁止
1.3 设计目标
完整的运行记录：持续记录设备硬件状态，覆盖全生命周期

故障可回溯：设备故障后可回溯至故障前状态

安全事件可追溯：安全事件可追溯到具体时间点

硬件退化可观测：长期记录可反映硬件退化趋势

数据主权保障：数据最终归用户所有（通过Core Cloud归档）

防篡改：除系统级调用外，任何读写行为被禁止

二、硬件规格
2.1 EEPRAM 阵列配置
参数	规格
阵列组成	16×1GB + 2GB（ECC）
总容量	18GB
单元类型	EEPROM（电可擦除可编程只读存储器）
ECC区域	2GB独立ECC区域，用于关键状态数据
写入寿命	≥100,000次（典型值）
数据保持	≥10年（典型值）
接口	专用总线（由Core RTOS管理）
2.2 存储分配
区域	容量	用途
主数据区	16GB	常规运行数据存储
ECC保护区	2GB	信任链状态、安全事件日志等关键数据
预留区	—	固件管理信息、磨损均衡表
三、数据记录内容
3.1 记录频率与分类
类别	记录频率	数据量（预估）	说明
基础设备信息	每月一次	2KB	设备型号、固件版本、硬件规格快照
运行状态	每小时一次	8KB	温度、频率、功耗、负载
电池健康	每小时一次	1KB	电压、容量、充放电状态、损耗
存储状态	每小时一次	0.5KB	可用空间、读写错误计数
安全事件摘要	每小时一次	1KB	CSMV状态、安全事件计数
异常快照	事件触发	5-10KB	系统崩溃、温度过高等异常时的详细记录
3.2 运行状态记录项（每小时）
数据项	说明	来源
时间戳	记录时间	Core RTOS系统时钟
CPU温度	各核心温度、封装温度	硬件传感器
CPU频率	各核心实时频率	CPU性能计数器
CPU功耗	封装功耗、各核心功耗	CPU功耗寄存器
CPU负载	各核心利用率	系统调度器
GPU温度	热点温度、全局温度	GPU传感器
GPU频率	核心频率、显存频率	GPU状态寄存器
GPU功耗	整卡功耗	GPU功耗寄存器
GPU负载	核心利用率、显存利用率	GPU性能计数器
电池电压	当前电压	电池监测电路
电池容量	当前容量、满充容量	电池管理芯片
充放电功率	当前充/放电功率	电池管理芯片
电池损耗	当前损耗百分比	电池管理芯片
内存使用率	已用/总量	系统内存管理
磁盘可用空间	各分区可用空间	文件系统
磁盘读写错误计数	SMART数据	磁盘控制器
CSMV状态	当前环境可信分	CSMV验证客户端
安全事件计数	累计安全事件数	Core RTOS安全模块
3.3 异常快照触发条件
触发条件	触发阈值	额外记录内容
温度过温	CPU/GPU ≥ 95°C	瞬时功率、散热状态、环境温度
温度过低	设备温度 ≤ -10°C	电池状态、加热器状态（如有）
系统崩溃	内核恐慌/蓝屏	崩溃前5分钟详细状态、崩溃现场信息
供电异常	电压跌落 ≥15%	电池放电曲线、供电转换记录
安全事件	检测到威胁	事件类型、触发时间、相关进程
ECC错误	内存/存储ECC纠错	错误地址、错误类型、累计次数
四、容量与寿命计算
4.1 总写入次数
参数	数值
记录周期	10年
小时数	10 × 365 × 24 = 87,600次
单次记录平均大小	<10KB
总数据量	<876MB
4.2 容量余量
区域	容量	实际使用量	余量
主数据区（16GB）	16,384MB	<876MB	>18倍余量
ECC保护区（2GB）	2,048MB	<100MB	>20倍余量
4.3 写入寿命
每小时一次写入频率远低于EEPROM的寿命上限（典型值≥100,000次）

即使考虑磨损均衡，EEPRAM阵列在10年使用周期内远未达到寿命终点

EEPROM寿命的终点在十年后更换时仍然远未触及，因此更换周期由业务规则定义，而非物理限制

五、写入与访问权限
5.1 写入权限
权限	说明
系统级调用	CoreOS / Core RTOS 可写入
内核空间	只允许内核级驱动调用
用户空间	禁止写入
第三方应用	禁止写入
5.2 读取权限
权限	说明
系统级调用	CoreOS / Core RTOS 可在必要时读取
用户空间	禁止读取
第三方应用	禁止读取
诊断模式	仅限授权服务人员通过专用工具读取
5.3 安全机制
EEPRAM阵列的读写指令仅由Core RTOS的驱动池管理器发出

任何非系统级调用尝试读取或写入EEPRAM，均被Core RTOS安全模块拦截并记录为安全事件

用户空间的任何读写请求将被忽略并记录

异常读写尝试会触发安全告警

六、十年更换周期
6.1 基本规则
规则	说明
标准周期	满10年必须更换
提前更换	可在1-9年申请提前更换
延期请求	特殊情况可通过Core支持申请延期（需经CPT审批）
6.2 更换流程
text
第1步：用户/售后确定需要更换
   │
   ▼
第2步：用户登录Core支持网站 → 申请更换 → 获取一次性令牌
   │
   ▼
第3步：携带设备 + 令牌 → 前往Core Dealer / Core Support Center
   │
   ▼
第4步：工作人员验证令牌 + 设备序列号 + 用户身份
   │
   ▼
第5步：在Core记录仪监控下进行操作
   │
   ├── 5.1 读取旧阵列数据，确认已上传Core Cloud
   │
   ├── 5.2 取出旧阵列
   │
   ├── 5.3 旧阵列进行无害化回收与销毁
   │   │
   │   ├── 物理销毁（确保数据不可恢复）
   │   ├── 记录销毁时间、方式、操作人
   │   └── 生成销毁证明（存入用户Core账户）
   │
   ├── 5.4 安装新阵列
   │
   └── 5.5 写入确认信息：
          “此EEPRAM阵列经过更换，原有数据存放在Core Cloud”

   │
   ▼
第6步：设备归还用户
   │
   └── 用户可在Core账户中查看：更换记录 + 销毁证明 + 云存档位置
6.3 令牌机制
属性	说明
有效期	30天
作用	验证申请真实有效
防篡改	绑定设备序列号、申请人身份、申请时间
可追溯	每次更换都有唯一记录
七、旧阵列处置与数据归档
7.1 数据归档
步骤	说明
1. 读取确认	更换前确认旧阵列数据完整
2. 上传Core Cloud	全部数据上传至用户Core Cloud账户
3. 验证上传	确认云上数据完整且可读
4. 生成云存档	生成云存档路径和时间戳
7.2 旧阵列销毁
旧阵列数据上传完成后，进行无害化回收与物理销毁

销毁方式：物理压碎/粉碎，确保不可恢复

销毁过程在Core记录仪下完成

销毁证明：包含销毁时间、方式、操作人，与云存档关联

7.3 用户可查阅
用户可通过Core账户查看：

设备所有历史运行记录（在线查看）

更换记录（每次更换的时间、地点、操作人）

销毁证明（关联每次更换的旧阵列销毁记录）

八、数据归属与隐私
维度	说明
数据所有权	数据属于用户
数据存储	本地 + Core Cloud（更换时归档）
数据访问	用户可通过Core账户查看历史数据
数据删除	用户可申请删除Core Cloud上的历史数据
第三方访问	未经用户同意，任何第三方不得访问
九、与Core生态的协同
9.1 与Core RTOS的关系
Core RTOS负责EEPRAM阵列的初始化、监控和管理

驱动池管理器负责执行写入操作

Core RTOS安全模块负责拦截非法读写尝试

9.2 与Core Cloud的关系
更换时，旧阵列数据自动上传Core Cloud

Core Cloud提供历史运行数据的永久归档

用户可在Core Cloud上查看设备完整生命周期状态

9.3 与CSMV的关系
CSMV验证结果和信任状态写入EEPRAM

提供跨平台环境可信度的可追溯历史

为安全审计提供证据链

9.4 与CoreSecuritySoftware的关系
安全事件由CoreSecuritySoftware产生

EEPRAM记录了安全事件的历史数据

恶意行为追溯时可以调取EEPRAM记录进行审计
