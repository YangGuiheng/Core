一、概述
1.1 定义
CSMV（Core Security Module Verification，核心安全模块验证）是Core生态定义的一套可信安全环境验证标准。它不要求用户安装特定品牌的安全软件，而是定义一套公开的验证规范，任何安全软件均可接入，通过验证后即可作为Core生态认可的“可信安全锚点”。

1.2 核心定位
属性	规格
名称	Core Security Module Verification（CSMV）
类型	安全环境验证标准
定位	Windows平台的信任锚点（对标macOS的SIP检测机制）
设计原则	不强制、可验证、去中心化、生态共治
1.3 设计目标
在缺乏统一系统级信任锚的平台（如Windows）上，构建生态级的可信环境验证机制

允许任何安全软件通过接入CSMV标准成为“可信锚点”

为应用提供统一的“设备可信度”查询接口

通过“限制模式”+“软标记”机制，实现对非可信环境的差异化服务策略

1.4 设计背景
在macOS平台上，Core生态通过检测SIP（系统完整性保护）状态来判断设备是否处于可信环境——SIP是苹果官方提供的统一信任锚点。

但在Windows平台上，不存在类似的单一系统级锚点。不同用户安装不同的安全软件，系统本身的信任状态难以用一个统一的系统级标志来衡量。CSMV正是为解决此问题而设计：通过构建一个安全软件生态联盟，让任何一款接入标准的安全软件都可以成为该设备上的“可信锚点”，从而为应用提供统一的可信环境判断依据。

二、CSMV 验证标准
2.1 验证维度
CSMV对安全软件的验证涵盖以下六个维度：

验证项	检查内容	目的
进程存在	安全软件的核心进程是否在运行	确保保护层在岗
服务状态	安全服务是否已启动且未禁用	确保保护层在工作
驱动加载	内核驱动是否已加载	确保保护层在底层
AMSI注册	是否注册到Windows反恶意软件接口	确保扫描能力可用
特征库更新	病毒库是否在合理时间内更新	确保不是摆设
签名有效性	安全程序的文件签名是否有效	确保未被篡改
2.2 验证方式
CSMV采用本地检测 + 云端验证的双重机制：

层级	方式	说明
本地检测	客户端SDK检测安全软件状态	实时采集进程、服务、驱动等状态信息
云端验证	CSMV云端服务器进行二次确认	验证安全软件签名、版本、信誉状态
2.3 验证结果
结果	含义	后续行为
通过	设备上存在至少一款通过CSMV验证的安全软件且正常运行	设备被标记为“可信”，可享受完整服务
未通过	无符合CSMV标准的安全软件在运行	设备被标记为“未验证”，服务受限
三、CSMV 状态码
状态码	名称	说明
CSMV_OK	验证通过	存在正常运行且已接入CSMV的安全软件
CSMV_NO_MODULE	无安全模块	未检测到任何已接入CSMV的安全软件
CSMV_MODULE_STOPPED	安全模块已停止	检测到CSMV安全软件进程存在，但服务未运行
CSMV_DRIVER_UNLOADED	驱动未加载	安全软件驱动未加载或已被卸载
CSMV_SIGNATURE_INVALID	签名无效	安全软件文件签名验证失败，可能被篡改
CSMV_OUTDATED	特征库过期	安全软件特征库未及时更新
CSMV_ERROR	验证异常	验证过程发生未知错误
CSMV_REVOKED	已吊销	该安全软件已被CSMV生态移除
四、CSMV 与 CoreSecuritySoftware 的关系
CSMV与CoreSecuritySoftware在Core安全体系中各司其职、互为补充：

维度	CSMV	CoreSecuritySoftware
定位	安全环境验证标准	安全能力执行引擎
作用对象	第三方安全软件生态	CoreOS原生安全系统
适用平台	Windows（无统一系统锚点）	CoreOS（原生微内核锚点）
核心功能	验证设备是否处于可信安全环境	提供蜜罐、行为分析、沙箱等深度安全能力
信任锚点	生态级（多厂商联盟）	系统级（CoreOS微内核）
输出结果	设备“可信/未验证”二值状态	多维安全事件与响应
两者协同关系：

在CoreOS平台上，由CoreSecuritySoftware直接提供安全服务

在第三方平台（如Windows）上，通过CSMV验证设备是否有可靠的安全软件守护

无论哪种情况，设备可信状态的最终判断都记录在云端，与账号和设备指纹绑定

五、CSMV 接入指南
5.1 安全软件接入CSMV
安全软件厂商可通过以下方式接入CSMV生态：

提交接入申请：向Core Foundation提交CSMV接入申请

技术审核：Core技术委员会对安全软件进行技术审查

实现标准接口：按CSMV规范实现状态查询接口

签署生态协议：同意遵循CSMV生态公约

发布认证版本：通过认证后，软件可标注“CSMV Certified”标识

5.2 CSMV对接接口
安全软件需实现以下标准接口，供CSMV客户端SDK调用：

c
// 查询安全软件状态
CSMV_Status CSMV_QueryStatus();

// 获取安全软件版本信息
CSMV_Info CSMV_GetModuleInfo();

// 报告威胁事件（可选）
void CSMV_ReportThreat(ThreatEvent event);
5.3 应用接入CSMV
应用开发者可通过CSMV客户端SDK接入：

c
// 查询当前设备是否通过CSMV验证
CSMV_Status status = CSMV_CheckEnvironment();

if (status == CSMV_OK) {
    // 设备可信，提供完整服务
    StartFullExperience();
} else {
    // 设备未验证，进入限制模式
    StartRestrictedMode();
}
六、限制模式与软标记
6.1 限制模式
当设备未通过CSMV验证时，应用可进入限制模式：

限制类型	可能的表现
视觉标记	应用/游戏画面四角显示“限制模式”水印，无法去除
功能受限	无法参与排位赛、天梯赛等竞技模式
收益打折	经验值、游戏币获取比例降低
社交限制	无法添加好友、无法加入战队
交易限制	无法进行任何形式的交易或赠送
6.2 账号软标记
在限制模式下使用应用/游戏，账号会被打上“软标记”

软标记是临时、可消除的嫌疑标记

消除方式：在通过CSMV验证的正常设备上，累计使用应用/游戏满半日

采用累计制，不要求连续使用

必须非空登录，有实际操作，非挂机

6.3 软标记消除流程
text
用户使用未通过CSMV验证的设备
    ↓
进入限制模式，账号被标记（软标记）
    ↓
用户切换到通过CSMV验证的正常设备
    ↓
正常使用应用/游戏，累计满半日
    ↓
软标记自动消除
6.4 设计意义
维度	说明
给用户“洗白”机会	不永久惩罚未通过CSMV的用户，只需切换到正常设备即可消除
提高作弊成本	批量作弊账号需要在正常设备上“养号”半日才能洗白
产生行为数据	半日观察期为服务器提供了充足的行为分析窗口
心理威慑	明确告知用户“你的账号正在被监控”，降低作弊意愿
七、CSMV 的跨平台意义
CSMV的核心理念具有跨平台价值，在不同操作系统上可适配不同的底层检测机制：

平台	信任锚点	实现方式
macOS	苹果官方SIP	检测SIP状态（系统级信任锚）
Windows	CSMV生态	验证通过CSMV认证的安全软件（生态级信任锚）
CoreOS	CoreOS微内核	原生集成CoreSecuritySoftware（原生级信任锚）
Linux	CSMV生态	验证通过CSMV认证的安全软件（生态级信任锚）
CSMV填补了Windows和Linux平台上“无统一系统级信任锚”的空白，通过生态联盟的方式构建了一个去中心化的可信环境验证网络。

八、CSMV 生态的未来演进
阶段	目标	说明
短期	建立标准	完善CSMV规范，推动首批安全软件接入
中期	形成生态	接入更多安全软件，建立CSMV认证品牌
长期	行业标准	推动CSMV成为跨平台可信环境验证的事实标准
九、结语
CSMV（Core Security Module Verification）是Core生态在缺乏统一系统级信任锚的平台上构建的生态级信任验证标准。它不强制用户安装特定安全软件，而是通过一套公开、透明、可验证的标准，构建了一个去中心化的可信安全环境网络。

不强制，但可验证；不统一，但可信。

这正是Core生态“生态共治”理念在安全领域的体现。
