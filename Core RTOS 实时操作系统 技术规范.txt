一、概述

1.1 定义

Core RTOS（Core Real-Time Operating System）是Core生态的底层操作系统，存在于硬件与所有上层操作系统之间。它由两个层面构成：

硬件固化层：与Core芯片深度集成，存在于物理硬件中，不可更改、不可卸载
软件运行层：运行在硬件固化层之上，用户可配置、可更新
Core RTOS是Core设备启动后加载的第一个软件，负责硬件初始化、多OS容器管理、算力资源分配与底层安全监控。

1.2 核心定位

属性	说明
全称	Core Real-Time Operating System
简称	Core RTOS
启动顺序	永远最先启动（硬件初始化后第一个加载的软件）
运行状态	始终在后台运行（包括关机态保持在线模式）
职责	硬件抽象、多OS管理、算力调度、安全监控、更新管理、关机态数据同步
设计哲学	不喧宾夺主——Core RTOS是舞台，不是演员
1.3 两个层面的关系

层面	性质	用户控制权	存储位置
RCRTOS（硬件固化层）	固化在硬件中，与Core芯片深度集成	不可更改、不可卸载	芯片内嵌ROM / 物理只读存储
VCRTOS（软件运行层）	运行在RCRTOS之上的软件逻辑	用户可配置、可更新	可写存储区域
RCRTOS是VCRTOS存在的基础。 VCRTOS不能脱离RCRTOS独立运行，但RCRTOS可以在没有VCRTOS的情况下维持设备的最低运行状态。

1.4 与 CoreOS 的关系

对比项	CoreOS	Core RTOS
定位	用户操作系统	底层实时操作系统
用户界面	提供完整GUI	Core Launcher
应用生态	提供原生应用与兼容层	不提供应用
启动顺序	Core RTOS之后	最先启动
运行状态	用户交互时运行	始终在后台运行（含关机态）
更新频率	频繁（迭代）	RCRTOS极低 / VCRTOS常规
二、架构设计

2.1 双层架构：RCRTOS + VCRTOS

Core RTOS 采用硬件固化层 + 软件运行层的双层设计：

层级	全称	说明
RCRTOS	Real Core RTOS	硬件固化层，存在于芯片物理硬件中，只读，永远稳定运行
VCRTOS	Virtual Core RTOS	RCRTOS 之上的软件运行层，承载日常变更与用户配置
2.2 RCRTOS（硬件固化层）

RCRTOS是Core RTOS的物理基础，与Core芯片深度集成：

属性	说明
存储位置	芯片内嵌ROM / 物理只读存储
更新方式	仅通过Core Back经75%共识更新（芯片更换级别）
用户控制权	不可更改、不可卸载、不可绕过
职责	硬件时钟管理、中断控制、电源管理、安全启动锚点、信任链第一环、VCRTOS监控、极简数据块读写能力
运行状态	始终运行（即使设备处于“完全离线”状态，RCRTOS仍然存在于硬件中，等待唤醒）
2.3 VCRTOS（软件运行层）

VCRTOS是用户实际可感知和可配置的CRTOS部分：

属性	说明
存储位置	RCRTOS之上的可写存储区域
更新方式	常规更新（通过虚拟层迭代）
用户控制权	可配置、可更新、可选择启用/停用部分功能
职责	多OS容器管理、算力调度、CL/CC通信、关机态数据同步、更新管理、Core Launcher
运行状态	跟随设备电源状态：运行中→全功能；关机保持在线→最低功耗运行；完全离线→停止运行

三、Core Launcher

实时硬件状态监控
启动链可视化展示
可启动操作系统列表与管理
系统日志查看
电源控制（关机/重启）

四、启动流程

4.1 启动顺序

Core RTOS 遵循 “永远最先启动” 的铁律：

text
上电 → 硬件初始化 → RCRTOS（硬件固化层）激活 →
  → 验证自身完整性 → 加载VCRTOS（软件运行层）→
    → 初始化 CLS/CLSDN 网络 → 向 CNS 注册设备 →
      → 加载 Core Launcher → 用户选择OS / 自动加载默认OS
4.2 启动时间

阶段	耗时（预估）	说明
上电，RCRTOS激活	<10ms	硬件固化层，无需加载
RCRTOS验证自身完整性	<20ms	信任链建立
加载VCRTOS	<50ms	软件层加载
初始化网络与向CNS注册	<100ms	CLS/CLSDN协议栈加载
加载Core Launcher	<50ms	用户界面显示
加载用户默认OS	取决于OS	Windows/Linux/Android等
五、关机状态行为

5.1 电源选项

用户在关机时可选择两种模式：

模式	行为	RCRTOS状态	VCRTOS状态
保持在线	设备关机，VCRTOS以最低功耗运行	始终运行	最低功耗运行
完全离线	设备完全断电，VCRTOS停止运行	仍存在于硬件中（等待唤醒）	停止
5.2 保持在线模式

设备选择“保持在线”关机后：

RCRTOS维持运行（始终在线）
VCRTOS以最低功耗模式运行
绝大部分外围硬件关闭
仅保留以下最小硬件：

CPU最低功耗核心（1-2核）
极小容量内存（CRTOS + 增量缓存）
存储控制器（数据块写入）
网络接口（CL/CC通信）
电源管理单元
功耗目标：< 0.5W

5.3 完全离线模式

设备选择“完全离线”关机后：

VCRTOS停止运行
所有硬件完全断电
RCRTOS仍存在于硬件中，等待下一次唤醒
下次开机时，RCRTOS首先激活，验证完整性，然后加载VCRTOS
VCRTOS在启动阶段检测离线期间的更新缺失，主动请求补发
5.4 关机态数据同步

在“保持在线”模式下，VCRTOS持续运行并承担数据同步职责（详见 DOC-2026-055）：

通过 Core Cloud 接收增量更新
通过 CL（CoreLink）进行本地中继
数据块级写入 SSD
冲突标记，等待用户启动后处理
六、多OS管理

6.1 容器化设计

Core RTOS 将所有上层操作系统视为 “容器”：

容器类型	说明	硬件访问方式
Windows	运行在 x64 能效核上（原生执行）	硬件直通
Linux	运行在 ARM 性能核或能效核上	硬件直通
Android	运行在 ARM 能效核上	硬件直通
CoreOS	运行在 ARM 性能核上	硬件直通
6.2 硬件直通

CPU核心可直通给任意OS（按需分配）
GPU可直通给当前正在渲染的OS
NPU可直通给需要AI推理的OS
存储通过RCRTOS代理访问（隔离且安全）
6.3 多OS切换

维度	传统双系统	Core RTOS 多OS管理
切换方式	重启，选择分区	无需重启，OS容器化切换
切换延迟	分钟级	毫秒级
底层控制	BIOS → 选中的OS	Core RTOS始终在场
安全隔离	一个OS中毒可能感染另一个	Core RTOS隔离，互不影响
七、更新机制

7.1 RCRTOS（硬件固化层）更新

几乎不更新
仅通过 Core Back 经75%共识更新
等同于更换芯片物理内容
7.2 VCRTOS（软件运行层）更新

遵循VCRTOS虚拟映射机制
更新预备 → 虚拟映射 → 24小时验证 → 用户选择 → 固化
八、安全模型

8.1 信任链

text
CSM（硬件信任根）→ RCRTOS（硬件固化层）→ VCRTOS（软件运行层）→ 上层OS
RCRTOS是不可篡改的信任链第一环
VCRTOS由RCRTOS验证和监控
上层OS无法绕过RCRTOS访问硬件
8.2 恶意软件防护

攻击方式	传统系统的结果	Core RTOS 的结果
修改系统文件	系统被破坏	改的是VCRTOS，丢弃即可恢复
安装内核驱动	系统可能蓝屏	驱动跑在VCRTOS，RCRTOS不受影响
持久化驻留	写入注册表/启动项	驻留在VCRTOS，RCRTOS无感知
提权到Ring0	可能成功	最高权限仍在RCRTOS掌控下
感染固件	极难清除	RCRTOS只读，无法被感染
8.3 蜜罐与行为分析

蜜罐管理员：运行在RCRTOS，监控所有VCRTOS的请求
行为分析引擎：运行在RCRTOS + CAI，分析VCRTOS行为
无重启热修复：RCRTOS替换VCRTOS中的恶意模块
九、与 Core 生态的协同

9.1 与玄央 APU 的关系

玄央特性	Core RTOS 利用方式
三类核心（ARM-P/ARM-E/x64-E）	不同类型OS分配到最优核心
50核心冗余	后台任务完全不影响前台用户体验
CSM硬件安全模块	信任链的第一环
9.2 与 CNS 的关系

Core RTOS启动后主动向CNS注册设备
CNS可唤醒处于休眠状态的Core RTOS设备
CNS通过Core RTOS管理设备的算力提供/消费状态
9.3 与 CSMV 的关系

Core RTOS内置CSMV验证客户端
在任何平台上，Core RTOS都可作为CSMV的信任锚点
9.4 与关机态同步机制的关系

VCRTOS在关机保持在线模式下承担数据同步代理职责
具体机制定义于 DOC-2026-055
