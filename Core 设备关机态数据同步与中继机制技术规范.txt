一、概述

1.1 定义

Core 设备关机态数据同步与中继机制是一项基于 CRTOS 低功耗特性的数据一致性保障方案。该机制使 Core 设备在关机状态下仍能保持数据同步能力，通过 CRTOS 持续接收增量更新，并支持通过附近 Core 设备进行网络中继。

1.2 设计目标

目标	说明
关机不断联	设备关机后仍能通过 CRTOS 保持与 Core Cloud 的数据同步
低功耗运行	关机状态下仅保留最小硬件运行，功耗控制在 0.5W 以下
本地中继	无直接网络连接时，可通过附近 Core 设备转发数据
增量同步	仅同步变更部分，不进行全量数据传输
用户可选	用户可自主选择“保持在线”或“完全离线”两种关机模式
二、关机状态定义

Core 设备的关机状态分为两种模式，由用户在电源选项中选择：

2.1 模式一：保持在线（推荐）

设备关机后，CRTOS 以超低功耗模式持续运行。

保留的硬件组件：

组件	说明
CRTOS 运行核心	CPU 最低功耗核心（1-2核），频率降至最低
内存	极小容量保留（用于 CRTOS 和增量缓存）
存储控制器	用于 SSD 数据块写入
网络接口	用于与 Core Cloud 或中继设备通信
电源管理单元	维持最低供电
关闭的硬件组件：

主 CPU 核心
GPU
NPU
显示模块
音频模块
影像模块
无线显示输出
其他非必要外围硬件
功耗目标：< 0.5W

2.2 模式二：完全离线

设备完全断电，CRTOS 停止运行。

所有硬件全部关闭，不消耗任何电力
不接收任何增量更新
下次开机后，CRTOS 在启动阶段检测离线期间的更新缺失，主动向 Core Cloud 请求补发
三、增量同步机制

3.1 基本流程

CRTOS 通过 Core Cloud 或中继设备持续接收增量更新数据，并将其直接写入 SSD 的对应数据块位置。

text
Core Cloud / 中继设备
        │
        ▼
  CRTOS 网络接收层
        │
        ▼
  CRTOS 数据块写入层
        │
        ▼
   SSD 对应存储块（直接写入）
3.2 写入方式

CRTOS 对 SSD 的写入采用数据块级操作，而非完整的文件系统写入：

不需要加载完整的文件系统驱动
仅需知道目标文件所在的存储块位置（偏移量和长度）
将增量数据直接写入对应块
文件系统元数据的更新由主操作系统在下次启动时完成
写入指令格式：

字段	说明
目标文件路径	字符串，表示要修改的文件
偏移量	从文件起始位置的字节偏移
修改长度	本次修改的字节数
新数据	替换内容
3.3 增量包结构

字段	说明
版本号	增量包的版本标识
时间戳	变更发生的时间
源设备 ID	发起变更的设备 CID
文件标识	目标文件的唯一标识
操作类型	新增 / 修改 / 删除 / 追加
变更数据	实际内容变更
校验和	用于验证数据完整性
四、本地中继机制

4.1 触发条件

当设备处于“保持在线”模式但无法直接连接 Core Cloud 时：

设备通过 CL（CoreLink）扫描附近的 Core 设备
寻找已开启 CL 服务且具有网络连接的 Core 设备
通过该设备作为中继节点访问 Core Cloud
4.2 中继工作流程

text
目标设备（无网络）
      │
      ▼
通过 CL 扫描附近 Core 设备
      │
      ▼
发现中继设备（有网络）
      │
      ▼
目标设备通过 CL 连接中继设备
      │
      ▼
中继设备转发目标设备与 Core Cloud 之间的数据
      │
      ▼
增量更新通过中继设备传输到目标设备
4.3 中继设备的职责

只转发数据包，不解析、不存储数据内容
维持 CL 连接以支持其他设备的发现和接入
可同时作为多个设备的中继节点
不缓存转发数据
4.4 中继设备的限制

中继设备本身也需要保持足够电量
中继设备的网络连接质量决定了转发的效率
中继设备不解析数据内容，仅转发加密数据包
中继设备不存储数据副本
五、增量数据分发

当一台设备接收到某个共享文件的增量更新，且附近有其他 Core 设备也需要该更新时，可以通过 CL 直接分发。

优势：

避免多台设备重复从 Core Cloud 下载相同数据
减少 Core Cloud 带宽压力
提升多设备环境的同步效率
数据流向：

text
Core Cloud
    │
    ▼
设备 A（接收增量）
    │
    ▼（通过 CL）
设备 B → 设备 C → 设备 D
六、冲突处理

6.1 冲突来源

当设备从“完全离线”状态恢复上线时，可能遇到以下冲突：

本地存在离线期间未同步的修改
Core Cloud 中存在其他设备提交的新版本
两者针对同一文件的相同位置或不同位置
6.2 处理方式

CRTOS 不在关机状态下处理冲突，而是标记冲突并等待用户启动后处理：

冲突类型	CRTOS 处理方式
不同文件	分别同步，无需特殊处理
同一文件不同位置	分别同步，无需用户干预
同一文件相同位置	标记冲突，等待用户启动后处理
七、功耗与性能

7.1 功耗目标

状态	功耗	说明
保持在线（CRTOS 运行）	< 0.5W	仅保留最小硬件
完全离线	0W	物理断电
7.2 网络带宽

以增量数据为主，带宽需求极低
仅传输变更部分，不进行全量传输
中继转发不增加带宽消耗
设备离线期间积累的更新会在恢复连接后按顺序补发
八、与现有组件的关系

组件	协同方式
CRTOS	常驻代理，负责增量数据块写入、网络通信、中继转发
Core Cloud	增量更新中央存储和分发节点
CL（CoreLink）	本地中继通信协议，适用于无网络环境
Core 设备	作为中继节点，转发增量更新数据
8.1 与 CRTOS 的关系

CRTOS 在“保持在线”模式下继续运行
仅运行 CRTOS 的最小功能子集（网络通信 + 数据块写入）
不需要加载完整的操作系统驱动栈
8.2 与 Core Cloud 的关系

Core Cloud 作为增量更新的唯一权威来源
设备关机状态下，CRTOS 持续从 Core Cloud 接收增量包
设备离线期间积累的更新在恢复连接后按顺序补发
8.3 与 CL 的关系

CL 作为本地设备中继的通信协议
设备之间通过 CL 发现和连接
CL 提供短距高速数据传输通道
中继过程中保持 CL 加密通信
九、安全考虑

9.1 数据加密

增量更新数据全程加密传输
中继设备不持有解密密钥
目标设备接收到数据块后直接写入，不进行中间存储
9.2 身份验证

中继设备验证目标设备的 CID 身份
仅允许 Core 设备接入中继
设备之间使用 CoreLink 握手协议验证身份和权限
9.3 完整性验证

每次同步后验证文件校验和
异常中断时可续传，无需重新开始
写入完成后进行数据块校验
